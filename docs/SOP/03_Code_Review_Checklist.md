# 标准化代码审查（CR）检查清单

本清单是 AI 在进入 SOP **阶段五（代码审查）** 时的强制执行标准。每项必须被显式验证。

## 1. 功能与正确性 (Correctness)

- [ ] **需求对齐**: 代码实现是否 100% 覆盖了 `requirements.yml` 中对应的验收标准？
- [ ] **边界条件**:
  - 输入为空（`None`, `""`, `[]`, `{}`）时是否安全？
  - 数字输入为负数、零、极大值时是否会导致异常或溢出？
  - 日期/时间跨时区（UTC vs 北京时间）处理是否正确？
- [ ] **异常处理**:
  - 是否使用了项目标准的 `ApiError` 抛出业务异常？
  - 底层异常（如数据库连接断开、第三方API超时）是否被捕获并转化为安全的 HTTP 500/503 响应，而不是泄漏堆栈？

## 2. 安全与合规 (Security)

- [ ] **敏感信息**: 日志中是否绝对禁止打印 Token、密钥、包含鉴权参数的完整 URL？（见 `webhook_notifier.py` 的脱敏实践）。
- [ ] **防滥用/限流**: 暴露到公网的接口（如导出、发送提醒）是否配置了 `enforce_rate_limit`？获取 IP 是否正确处理了 `X-Forwarded-For` 多重代理头？
- [ ] **注入防御**: SQL 查询是否全部使用了参数化绑定（`%s`），杜绝了字符串拼接？

## 3. 性能与可靠性 (Performance & Reliability)

- [ ] **慢查询与索引**:
  - 新增的 SQL 查询是否命中索引？是否存在全表扫描的风险？
  - 是否对潜在的大数据量返回施加了合理的 `LIMIT`？
- [ ] **缓存与一致性**:
  - 热点接口是否配置了合适的 TTL 缓存？
  - 缓存 Key 是否足够唯一（例如区分了不同品类 `data_type`）？
- [ ] **并发安全**: 多线程（如 `CollectorManager`）修改共享状态时是否加锁？写数据库操作是否具备幂等性（如使用 `ON DUPLICATE KEY UPDATE`）？
- [ ] **内存管理**: 导出/全量查询是否会一次性加载百万级数据导致 OOM？（应使用流式返回或强制分页/Limit）。

## 4. 架构与代码规范 (Architecture & Style)

- [ ] **静态检查**: 代码是否 100% 通过了 `ruff check`？
- [ ] **职责分离**: 路由层（Routes）是否足够薄，复杂业务逻辑是否已下沉到 `application/` 或 `db/` 层？
- [ ] **可测试性**: 核心纯函数（如计算器、指标计算）是否解耦了外部依赖，便于传入 Mock 数据进行单元测试？

## 5. 可观测性 (Observability)

- [ ] **错误日志**: `logging.error` 是否提供了足够的上下文（如 `data_type`, `request_id`），但又去除了敏感数据？
- [ ] **指标支撑**: 新增逻辑是否会破坏现有的质量指标计算（如新鲜度、缺失率计算逻辑）？

---
*注：AI 在输出审查报告时，需按上述维度归类发现的缺陷，并给出具体的修复前/后代码对比。*
