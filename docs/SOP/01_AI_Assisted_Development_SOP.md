# AI辅助开发全生命周期标准化流程（SOP）

**文档状态**: 已发布
**版本**: V1.0
**适用对象**: AI 编程助手、产品经理、研发工程师、测试工程师

---

## 1. 流程总览

本体系旨在规范 AI 辅助开发过程，确保输出的代码与文档“可重复、可追踪、可度量”。全生命周期划分为 6 个阶段：
`需求分析` → `技术设计` → `代码实现` → `测试验证` → `代码审查` → `部署发布`。

---

## 2. 阶段标准定义

### 阶段一：需求分析与拆解 (Requirement Analysis)

- **输入标准**: 用户的原始需求描述、会议记录或一句话需求。
- **AI 核心动作**:
  1. 调用 `SearchCodebase` 与 `Read` 检索现有仓库，梳理上下文。
  2. 运用产品经理视角进行需求合理性评估。
  3. 将高层业务目标拆解为 FR（功能需求）与 NFR（非功能需求）。
- **输出交付物**:
  - `docs/Product_Analysis.md`: 产品价值与合理性评估。
  - `docs/PRD.md`: 产品需求文档（含用例图、用户故事、指标定义）。
  - `docs/SRS/requirements.yml`: 结构化、唯一编号、可测试的需求规格定义。
- **质量门禁 (Quality Gate)**:
  - 需求颗粒度必须达到“可测试”级别。
  - 需求追踪矩阵（RTM）已建立并包含 `SourceRef`。

### 阶段二：技术方案设计 (Technical Design)

- **输入标准**: 已通过门禁的 PRD 与 SRS。
- **AI 核心动作**:
  1. 梳理数据库表结构变更、API 接口契约设计。
  2. 使用 Mermaid 编写 UML 图（时序图、状态图、组件图）。
  3. 定义异常处理规范与核心性能指标（SLO）支撑方案。
- **输出交付物**:
  - `docs/Technical_Architecture.md`: 架构与方案设计文档。
  - `docs/SRS/attachments/uml/*.mmd`: 流程与架构可视化图表。
  - API 契约（包含在架构文档或独立的 Swagger/OpenAPI 定义中）。
- **质量门禁**:
  - 架构设计必须符合仓库现有技术栈（Flask/React/MySQL）。
  - NFR（如限流、并发、安全）在设计中有明确的应对策略。

### 阶段三：代码实现 (Code Implementation)

- **输入标准**: 技术方案设计文档、API 契约。
- **AI 核心动作**:
  1. 优先使用 TDD（测试驱动开发）模式：先搭测试骨架，再写业务代码。
  2. 严格遵循项目中已确立的代码规范（如统一错误码 `ApiError`，类型注解 `typing`）。
  3. 修改代码后主动更新需求跟踪矩阵（RTM）的 `CodeRef`。
- **输出交付物**:
  - 业务源码文件（`src/`、`templates/` 等）。
- **质量门禁**:
  - 无冗余代码、无硬编码的敏感信息。
  - `ruff check src` 静态分析无报错。

### 阶段四：测试验证 (Testing & Verification)

- **输入标准**: 业务源码、SRS 中的验收标准。
- **AI 核心动作**:
  1. 针对新增逻辑编写单元测试与集成测试（使用 `pytest`）。
  2. 覆盖正常主干链路、边界条件（空值、极大值）及异常态（断网、限流）。
  3. 主动运行本地测试环境，验证接口返回与前端表现。
- **输出交付物**:
  - 测试用例文件（`tests/test_*.py`）。
  - 测试运行结果。
- **质量门禁**:
  - 新增代码的测试覆盖率应不低于 80%。
  - `pytest -q` 全量通过（0 failures）。

### 阶段五：代码审查 (Code Review)

- **输入标准**: 待合并的代码及测试用例。
- **AI 核心动作**:
  1. 基于预定义的《代码审查检查清单》进行交叉审视。
  2. 重点排查：并发安全、内存泄漏、边界条件、错误降级。
  3. 修复审查发现的缺陷，并输出修复报告。
- **输出交付物**:
  - `docs/bugfix/YYYY-MM-DD_code_review_and_fixes.md`: 审查与修复说明。
- **质量门禁**:
  - 清单项全部确认（Checklist passed）。
  - 致命/高危缺陷密度为 0。

### 阶段六：部署发布与回滚 (Deployment & Rollback)

- **输入标准**: 已通过 CR 的代码分支（通常为 `main`）。
- **AI 核心动作**:
  1. 确认 CI/CD 管道配置（如 `.github/workflows/python-app.yml`）状态。
  2. 编写发布 Changelog 并提交推送。
- **输出交付物**:
  - `CHANGELOG.md` 更新（可选）。
  - Git Commit & Push 记录。
- **质量门禁 (部署标准)**:
  - 镜像构建成功（`docker compose build`）。
  - 生产/准生产健康检查接口（`/api/health`）返回 200。
- **回滚机制**:
  - 优先支持代码级回滚：`git revert <commit_id>` 并重新推流。
  - 数据库变更回滚：在设计期必须提供向下兼容的 Schema 变更（不随意 Drop 列）。

---

## 3. 产物规范与追踪机制

为保证 100% 的可追踪性，本项目采用 **RTM（需求跟踪矩阵）** 驱动：

1. `requirements.yml` 是唯一事实来源。
2. 任何代码的提交必须能够回答：“这行代码在实现哪个 SRS 编号的需求？”
3. 使用自动化脚本 `tools/generate_srs.py` 将 YAML 转换为人类可读的 MD 和 Excel。
