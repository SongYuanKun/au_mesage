# AU Metal Price Platform｜产品分析评估（PM视角）

更新时间：2026-05-12

## 1. 项目一句话与范围

- 一句话：一个面向贵金属投资者与回收从业者的“多源价格采集 + 实时监控 + 历史分析 + 价格提醒”平台。
- 当前实现边界（以仓库代码为准）：
  - 后端：Flask Web + REST API；服务启动并行采集多个数据源并落 MySQL；热点接口进程内 TTL 缓存。
  - 前端：Jinja2 模板页面（首页/历史/分析）为可运行主链路；仓库内存在 React SPA 版页面与状态管理，但与主链路并存且未完全对齐。

## 2. 目标用户与场景清晰度评估

### 2.1 目标用户画像（建议明确为三类）

1) 个人投资者（轻中度）
- 目标：随时掌握金/银价格、判断买卖点、设置提醒。
- 关键场景：通勤/工作间隙快速看盘；触发目标价提醒；查看近 7 天/近 1 小时走势。

2) 回收/金店从业者（高频）
- 目标：快速对比大盘价与回收价/零售价，评估报价与毛利空间。
- 关键场景：门店报价；批量下单/回收前的快速计算；关注汇率对本币价格影响。

3) 数据型用户（量化/开发者/B2B）
- 目标：稳定 API、数据可追溯与质量可控，用于策略回测、数据看板或二次分发。
- 关键场景：拉取历史 OHLC；订阅提醒；对接内部系统。

结论：目标用户“可被清晰定义”，但当前产品文案/信息架构未明确区分三类用户的任务与诉求，导致卖点表达偏泛。

### 2.2 用户痛点匹配度

痛点 1：数据源分散且口径不一致
- 现状支持：多源采集与入库已具备；但缺少“口径说明/数据源权重/冲突处理策略”，用户仍可能不信任。

痛点 2：需要低成本的实时监控与提醒
- 现状支持：SSE 订阅 + 多通道通知已具备雏形，是明确价值点。

痛点 3：缺少面向“交易/回收”任务的快速计算
- 现状支持：价差计算器与历史对比 API 已具备，但入口与解释不足（缺少“结果可解释”与“参数模板”）。

结论：核心功能总体能覆盖痛点，但“可信度（数据质量/口径）”与“任务化体验（投资/门店/数据用户的各自路径）”仍是决定留存的关键。

## 3. 创新性与竞争优势评估

### 3.1 竞争环境（典型替代方案）

- 资讯/行情类 App 与网站：功能全面、覆盖品类多，但提醒、口径、数据导出、API 通常受限。
- 交易所/银行/平台自有行情：可信但口径固定、可定制与分析工具有限。
- 开发者抓取脚本：可控但维护成本高、稳定性差。

### 3.2 本项目的可形成优势点

1) “采集-存储-分析-提醒”闭环可控
- 与单纯展示行情相比，可把“提醒/导出/API”做成差异化。

2) 通知通道可扩展
- 企业微信/Telegram/邮件的多通道组合，更适合专业用户与 B2B 场景。

3) 轻量化部署
- Docker + MySQL 即可运行，利于私有化/自建。

### 3.3 当前不足（会削弱竞争力）

- 产品叙事与文档与真实实现存在偏差（例如会员/订阅、部分架构描述），会影响团队协作与对外可信度。
- 数据可信度体系缺失：缺少数据源健康度、延迟、缺失率、异常值识别与标注。
- 两套前端并存但边界不清：会拖慢迭代速度与体验一致性。

结论：创新性不是“从 0 到 1 的新模式”，而是“可控的数据闭环 + 专业提醒 + 任务型工具”。若补齐数据质量与体验路径，有机会在细分市场站稳。

## 4. 商业模式可持续性评估

### 4.1 可选商业模式

P0（最匹配当前形态）：私有化部署 / 定制交付
- 客户：金店/回收连锁、工作室、媒体小站、数据看板团队。
- 收费：部署费 + 维护费；数据源与通知通道按量。

P1：SaaS 订阅（面向专业用户）
- 分层：免费（基础看盘）/ 专业（提醒、导出、更多周期 OHLC、更多数据源）/ 企业（API、SLA、团队协作）。

P2：数据 API/数据包
- 以“稳定性/合规/可追溯”作为差异化，向开发者与机构售卖。

### 4.2 关键成本与风险

- 数据源成本：第三方 API 可能收费/限流；抓取站点存在封禁与法律风险。
- 基础设施成本：若做 SaaS，存储与计算、通知通道、监控告警均会成为刚性支出。
- 合规风险：对金融数据展示与传播的合规要求需评估。

结论：短期最可持续路径是“私有化/定制 + 专业提醒/分析”切入；SaaS 需先补齐数据质量与稳定性指标体系。

## 5. 用户体验设计成熟度评估

### 5.1 现有体验亮点

- 首页即呈现核心信息（实时价、涨跌、更新频率）+ 计算器/提醒入口，具备任务完成闭环。
- 历史/分析页提供趋势与比率图，形成“看盘-分析”的基本链路。

### 5.2 体验缺口（建议优先解决）

- 信息架构：不同用户任务混在同一页面，缺少“投资者模式/门店模式/数据模式”入口或预设。
- 解释体系：指标、口径、数据来源与延迟缺少显性说明；用户难以建立信任。
- 异常态：数据缺失、延迟、采集失败时的提示与降级缺失。
- 配置化：提醒条件、通知通道、刷新频率等缺少统一的设置中心。

结论：用户体验“可用但不够专业”。提升可信度与异常态体验，是提升留存与付费转化的核心。

## 6. 技术实现路径合理性评估

### 6.1 当前架构可行性

- Flask + MySQL + 多采集器并行：适合 MVP 与自建场景，成本低、迭代快。
- 进程内 TTL 缓存：适合热点接口的快速优化，但不适合多实例与高可用扩展。
- SSE 提醒：实现简单、浏览器端易用，但需要处理连接数与心跳、断线重连等问题。

### 6.2 风险点与建议演进

- 采集与 Web 同进程：采集阻塞/崩溃会影响 Web 服务，建议拆为独立采集服务（或作业进程）并引入队列/定时调度。
- 缓存与状态：若走 SaaS/多实例，建议引入 Redis（缓存 + 分布式锁 + 通知状态）替代进程内缓存。
- 数据质量：补充“采集任务表/数据源状态表/异常值标记字段”，并增加质量指标产出。
- 前端路线：建议明确“模板页为运营版，React 为下一代 SPA”，或二选一以降低维护成本。

结论：技术路径总体合理可行，但要支撑商业化与规模化需要在“进程解耦、可观测、数据质量、缓存一致性”上补课。

## 7. 综合判断（合理性/价值/潜力）

- 合理性：中高。需求存在，MVP 形态成立，但边界与文档需收敛。
- 用户价值：中高。提醒与任务型计算器是可感知价值；数据可信度提升后价值显著上升。
- 市场潜力：中。大众行情市场竞争激烈，但“专业提醒 + 私有化 + 可控数据”的细分市场有机会。

## 8. 产品方向建议（下一步的“关键抉择”）

1) 先明确主赛道：
- 若做 ToC 看盘：需要强运营、强品类扩展、强内容与社群，投入高。
- 若做 ToB/自建：优先做稳定、可配置、可审计、可扩展的采集与告警。

2) 把“可信度”当作第一增长杠杆：
- 每个价格展示旁标注数据源与更新时间；提供“数据健康状态”；异常值与缺失标记。

3) 收敛两套前端与文档：
- 以真实实现为准重写 PRD 与里程碑，避免团队协作成本。
