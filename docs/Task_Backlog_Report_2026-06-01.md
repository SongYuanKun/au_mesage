## 6. Top 3 高优任务：在最新代码环境下的可落地方案

### 6.1 FR-AUTH-001：认证与权限控制

**原始需求**：SRS `FR-AUTH-001`  
**建议实现策略（优先级从高到低）**：

1) **方案 A（推荐）**：Supabase Auth（前端）+ 后端仅做“鉴权校验 + RBAC”
- 原因：仓库前端已依赖 `@supabase/supabase-js`，更贴合既有技术方向。
- 后端：新增 `Authorization: Bearer <jwt>` 校验（JWKS 或 shared secret）+ 角色映射。

2) **方案 B**：纯后端 JWT（自建用户表 + 登录接口）
- 原因：依赖更少，但会增加安全与运维成本（密码存储、重置等）。

**实现步骤（方案 A）**：
- 前端：
  - 新增登录/注册页面或弹窗（或先使用 Supabase magic link）。
  - 在路由守卫处保护 `/alerts` 的“持久化配置功能”、未来的 `/admin/*` 页面。
- 后端：
  - 新增 `src/auth/`：token 校验与角色解析。
  - 为管理端 API、导出接口等加 `@require_role(...)`。
  - `docs/API.md` 补充认证与错误码。

**涉及文件（预计）**：
- 后端：`src/route.py`（注册 before_request）、`src/api_errors.py`、`src/routes/api/*`
- 前端：`src/App.tsx`、`src/pages/*`、`src/api/price.ts`
- 文档：`docs/API.md`、`docs/SRS/requirements.yml`（状态/链接）

**需要协调**：
- 产品：确定角色体系（访客/用户/管理员/运维）与最小权限集。
- 运维：确定 Supabase 项目/密钥管理与环境变量注入方式。

### 6.2 FR-ADM-001：数据源配置与优先级管理

**原始需求**：SRS `FR-ADM-001`  
**目标**：让管理员可启用/禁用数据源、设置主/备源与优先级，并可查看数据源健康度。

**实现步骤**：
- 数据层：
  - 新建配置表（建议 `data_source_config`）记录 `source_id`、`enabled`、`priority`、`updated_at` 等。
- 采集层：
  - `CollectorManager` 读取配置决定是否启动某采集器。
  - `get_price_overview_data` 的 `source` 输出与配置优先级对齐（必要时按优先级写入/展示）。
- 管理端：
  - 新增 `/admin/sources` 页面（前端）+ `/api/admin/sources`（后端）。

**风险点**：
- 配置误操作导致数据中断：需要“保存前校验 + 一键回滚到上一版本配置”。

### 6.3 SEC-002：审计日志与关键操作追踪

**原始需求**：SRS `SEC-002`  
**目标**：把“谁在什么时候做了什么配置/导出/登录”留下可追溯证据。

**实现步骤**：
- 新建审计表 `audit_log`（最小字段：actor、action、target、before/after、ip、user_agent、created_at）。
- 在以下行为埋点：
  - 登录/登出（FR-AUTH-001 先行）
  - 数据源配置变更（FR-ADM-001）
  - 导出接口调用（现有 `/api/export/history`）
- 提供查询接口（管理员/运维可用）：`GET /api/admin/audit?start=...&end=...&action=...`
- 日志脱敏：任何密钥、token、webhook URL 不落库明文。
